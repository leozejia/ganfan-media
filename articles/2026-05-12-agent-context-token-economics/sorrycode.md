# 如何高效利用上下文

这篇是早期候选长文。当前公开 docs 已经硬切为：

```text
/docs/agent-infra/context-management
```

后续不要直接发布这份长文。需要继续扩展时，优先在现有 `context-management` 页面上补充，或者另开一篇更具体的任务教程。

## 先理解上下文

Agent 的上下文不只是你刚刚打的那句话。

它可能包括：

- 系统指令
- 项目说明
- 工具定义
- 文件内容
- 聊天历史
- 错误日志
- 搜索结果
- 截图
- 任务列表
- 压缩摘要

这些内容会影响 agent 下一步怎么判断、怎么调用工具、怎么写文件。

所以高效使用 agent，不只是会提问，还要会管理上下文。

## 三类上下文

把上下文分成三类最容易上手。

### 长期规则

长期规则是会反复使用的稳定信息。

比如：

- 项目是什么
- 目录怎么分
- 测试命令是什么
- 代码风格是什么
- 品牌颜色是什么
- 不要改哪些文件
- 输出文件放在哪里

长期规则应该写进文件，不要每次都靠聊天重复。

常见文件：

```text
AGENTS.md
CLAUDE.md
DESIGN.md
README.md
docs/
```

### 当前任务

当前任务是这一次要完成的目标。

比如：

- 修改一个页面
- 写一篇文章
- 整理一份资料
- 生成一张封面
- 修一个报错
- 做一页 PPT

当前任务可以写进 `brief.md`、`TASK.md` 或当前会话里，但要短，要具体。

好的任务描述通常包括：

- 目标
- 输入材料
- 输出形式
- 不要做什么
- 怎么算完成

### 临时证据

临时证据是当前判断需要的材料。

比如：

- 一段错误日志
- 一张截图
- 一次搜索结果
- 一个失败方案
- 一版旧稿
- 一组临时数据

这些内容不应该永久留在上下文里。判断完成后，保留结论，删除噪音。

## 推荐工作流

### 1. 先让 agent 读项目

不要一上来就让 agent 改文件。

可以先说：

```text
先不要修改任何文件。请阅读这个项目的目录结构，告诉我它是做什么的、入口在哪里、我应该先看哪些文件。
```

如果是设计、文章、PPT，也可以换成：

```text
先不要生成最终稿。请阅读我给你的材料，告诉我当前目标、可用素材、缺失信息和你建议的执行顺序。
```

### 2. 把稳定规则写进文件

代码项目先写 `AGENTS.md`。

设计项目先写 `DESIGN.md`。

文章或运营项目先写 `brief.md` 和 `sources.md`。

不要把所有规则都塞进一条很长的提示词。提示词适合发起任务，文件适合沉淀规则。

### 3. 每完成一阶段就做检查点

长任务不要一路聊到底。

阶段结束后，让 agent 输出一份短交接：

```text
请把当前进展压缩成一份交接，保留：
- 当前目标
- 已确认的事实
- 已完成的修改
- 已废弃的方向
- 关键文件和链接
- 下一步只需要关注的内容
- 旧会话里不要再沿用的信息
```

把这份交接保存到任务文件里，或者用它开启新会话。

### 4. 发现上下文变脏就重开

出现这些情况，建议开新会话：

- 任务方向变了
- agent 开始引用旧结论
- 会话里有大量失败尝试和日志
- 你已经多次纠正同一个误解
- 你说不清当前会话应该以哪条指令为准

重开前不要直接丢掉历史。先让 agent 生成交接，再用交接开启新会话。

## 常见误区

### 误区一：资料越多越好

资料多不等于上下文好。

如果 agent 要先在一堆无关材料里找重点，它就同时在做检索和推理。更好的做法是先筛掉无关材料，只给和当前任务有关的部分。

### 误区二：长会话一定更省钱

长会话可能复用稳定上下文，也可能把无关历史一直带着跑。

缓存命中只能说明某段内容被重复利用，不能说明那段内容有价值。

### 误区三：把规则都写进聊天

聊天历史会越来越长，也容易混进临时判断。

稳定规则应该写进文件。文件能被审查、复用、更新，也更容易在不同 agent 之间迁移。

### 误区四：压缩就是删东西

好的压缩不是把字数变少，而是把当前任务需要的状态留下来。

应该保留：

- 决策
- 约束
- 当前目标
- 关键证据
- 下一步

应该删除：

- 重复日志
- 失败尝试细节
- 已废弃方案
- 无关聊天
- 过期判断

## 新手最小模板

如果你不知道从哪里开始，在项目根目录放一份 `AGENTS.md`：

```md
# AGENTS.md

## Project

这个项目是：

## Commands

- 安装：
- 开发：
- 测试：
- 构建：

## Rules

- 修改前先说明计划。
- 不要改动和任务无关的文件。
- 修改后说明改了哪些文件。

## Do Not

- 不要提交 API Key、密码或 token。
- 不要删除我没有明确要求删除的文件。
```

设计任务可以放一份 `DESIGN.md`：

```md
# DESIGN.md

## 项目气质

这个项目应该看起来：

## 颜色

- 主色：
- 辅助色：
- 避免：

## 排版

- 标题：
- 正文：
- 间距：

## 禁止事项

- 不要：
```

文章或运营任务可以放一份 `brief.md`：

```md
# Brief

## 目标

## 读者

## 核心问题

## 输入材料

## 输出形式

## 不要写什么

## 完成标准
```
