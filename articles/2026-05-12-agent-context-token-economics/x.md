# 上下文不是越多越好：Agent 时代的 Token 经济学

做 SorryCode 中转站的时候，我发现一个很有意思的变化。

很多老牌中转站，其实看不见细的数据统计。

以前我用这些站，习惯是差不多就把会话释放掉。

因为那个“年代”，会话长了就开始出现大量幻觉了，定期释放会话是常见的行为。

后来我开始长期用 Codex，从 gpt-5.3-codex 到 gpt-5.4，再到现在，我最大的感受是：会话压缩变得很强。

一个会话跑很久，细节也不怎么丢。以前我会担心：会话太长了，压缩一下是不是就忘了前面做过什么。现在很多时候不是，它能接上，能记得项目状态，能继续往下干。

这很容易让人上头。

说出来不怕大家笑话，SorryCode 现在其实没什么用户，平台大部分用量都是我自己烧的。

有一段时间，我一个人贡献了绝大多数用量，每天能跑掉很夸张的 token。

一开始我还挺兴奋。

我看 usage，缓存命中高得离谱，90% 以上，甚至 95% 都见过。

我当时的第一反应是：

```text
这不是省麻了吗？
```

直到有一天，我发现自己每天能烧出 100 多美刀。

我才开始认真看账单。

结果发现一件很扎心的事：我的 input 不多，output 也不多，大头全在 cached input。

也就是说，我确实命中了缓存，而且命中得很好，但它不是免费。

我去查了一下 OpenAI 的价格，gpt-5.5 的 cached input 还是 $0.50 / 1M tokens。

听起来很便宜。

问题是，当用量上来以后，便宜也会变贵。

这就是我最近对长会话改观的地方。

以前我会觉得：

```text
会话养起来了，缓存命中高，就是好事。
```

现在我不这么看了。

缓存命中高，只能说明一件事：

```text
有一段上下文被反复利用了。
```

它没告诉你那段上下文值不值得继续留。

如果里面是项目规则、当前架构、已经确认的事实，那是资产。

如果里面是过期判断、废弃方案、调试时塞进去的日志、agent 自己写错过的总结，那也是缓存。

只不过你是在便宜地重复一堆垃圾。

便宜，不等于划算。

这就是 Agent 时代的 Token 经济学。

它不是问一个 token 多少钱。

它问的是：

```text
这个 token 还在推进任务吗？
```

长会话当然有价值。我现在也不会一出问题就开新会话，因为 Codex 这种工具确实可以把一个复杂项目接很久，尤其是压缩做得好的时候。

但长会话不是养宠物。

不是越久越有感情。

会话越长，里面越可能同时存在两种东西：

```text
稳定资产
历史负债
```

稳定资产应该留下，历史负债应该清掉。

所以现在我管理长任务，会做一件事：定期释放会话。

不是直接关掉。

我会先让它做交接。

我一般会这么说：

```text
我准备释放这个会话了。

请你复核一下当前文档的健康程度，准备一份 handoff 给你的继任者。

这份 handoff 要包括：
- 当前目标
- 已确认的事实
- 已经废弃的方向
- 关键文件和链接
- 下一步只需要看的内容
- 哪些旧信息不要再用

最后给继任者一句话，让他能快速拉齐项目情况。
```

这句话很重要：

```text
哪些旧信息不要再用
```

很多 handoff 只写“我们做了什么”。

但真正救命的是“哪些东西已经作废”。

因为新 agent 最容易被旧结论污染。

你不给它标出来，它就会把历史里的每一句话都当成可参考材料。

这也是为什么我一直建议，把长期真相写进文件，而不是全塞在聊天里。

写代码，可以是 `AGENTS.md`。

做设计，可以是 `DESIGN.md`。

写文章、做运营、做产品，也可以是 `brief.md`、`sources.md`、`handoff.md`。

聊天记录适合推进，文件适合沉淀，这两个东西不能混。

很多人用 agent 的方式，其实是在把聊天记录当数据库。

越聊越长。

越长越舍不得删。

舍不得删，就开始迷信缓存。

缓存命中一高，就觉得自己赚了。

我现在觉得这很危险。

因为缓存只能帮你降低重复成本。

它不能帮你判断重复的东西是不是还该存在。

什么时候该释放会话？

我一般看四个信号：

```text
方向变了
agent 开始引用旧结论
会话里堆了大量日志和失败尝试
你已经说不清它现在到底该听哪条指令
```

出现这些信号，就不要再硬养。让它交接，把真的东西写进文件，把废掉的东西明确标废，然后开新会话。

这不是浪费上下文。

这是把上下文从垃圾堆整理成档案。

Agent 越强，这件事越重要。

因为强 agent 的问题不是干不了活。

它的问题是太能干了。

它能沿着一个脏状态继续干很久，干得很认真，也烧得很认真。

所以别只看缓存命中率。

也别只看会话能不能压缩。

真正要看的是：

```text
长期真相有没有沉淀
临时噪音有没有清掉
下一任 agent 能不能一眼接上
```

我把长会话、缓存、handoff 和 `AGENTS.md / CLAUDE.md / DESIGN.md` 这类上下文文件的入口整理在 SorryCode 这里。

不是让你多写文档。

是让你少把钱花在重复解释和重复背垃圾上。
