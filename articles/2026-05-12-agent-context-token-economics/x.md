# 上下文不是越多越好：Agent 时代的 Token 经济学

我现在判断一个人会不会用 agent，不看他会不会写提示词。

我先看他会不会清上下文。

很多人还在用聊天机器人的习惯用 agent：写文章，把十几份材料全丢进去；做 PPT，把前三版草稿、老板意见、自己推翻过的想法全留在会话里；做设计，把每一次不满意的图、每一句不要这样、每一个临时审美偏好都堆在历史里；改项目，让 agent 从读代码、报错、修 bug、改方向一路跑到第二天，中间从来不收拾。

看上去很努力，实际是在把桌面上的废纸一张张塞进它脑子里。

Agent 跟普通聊天不一样。它会带着上下文继续干活。上下文给得好，它接手的是一个清楚的工作台。上下文给得烂，它坐在一堆旧纸箱中间找钥匙。

这里最容易误判的一句话是：

```text
上下文越多，agent 越聪明。
```

不对。

上下文是资产，也是负债。

资产是什么？

项目规则、品牌风格、已确认的事实、当前目标、文件路径、验收标准。

这些东西稳定、可复用，能减少反复解释。

负债是什么？

已经放弃的方案、过期的错误日志、临时试错、被推翻的判断、前一版审美偏好、agent 自己写错但还留在历史里的总结。

这些东西继续留在会话里，不会显得资料充分，只会让 agent 每一轮多背一点旧账。

这就是 Agent 时代的 Token 经济学。它问的不是一个 token 多少钱，而是每个 token 在推进任务，还是在消耗模型注意力。

缓存也别神化。

OpenAI 和 Anthropic 的文档都讲得很清楚，prompt caching 主要解决的是重复上下文的成本和延迟。稳定前缀重复出现，模型可以更便宜、更快地处理。

但缓存不会判断那段上下文该不该留。即使一段没价值的历史反复出现在前缀里、命中了缓存，它也还是没价值。

所以别只追求长会话，也别只追求缓存命中。更重要的是把上下文分成三类：

```text
长期规则
当前任务
临时证据
```

长期规则应该进文件。

写代码，放进 `AGENTS.md` 或 `CLAUDE.md`。

做设计，放进 `DESIGN.md`。

写文章、做 PPT、做运营计划，放进 `brief.md`、`sources.md`、`TASK.md` 这类任务文件。

这些文件比聊天记录可靠。它们可以被检查、修改、复用，也不会因为一轮闲聊混进一堆无关内容。

当前任务要短。

不要让 agent 同时记住我要重构首页、顺便修登录、再研究增长、最后生成海报。任务一大，它就容易把所有东西都当相关。

给它一个清楚目标：

```text
现在只做这一步。
完成后告诉我：
1. 已确认什么
2. 排除了什么
3. 下一步需要什么
4. 哪些旧信息不要再用
```

临时证据要及时丢。

错误日志、搜索结果、截图、旧版本草稿，都只服务当前判断。判断做完，把结论留下，把原始噪音清掉。

长期任务最好的节奏不是一路聊到底，而是定期做检查点：

```text
把目前会话压缩成一份交接：
- 当前目标
- 已经确认的事实
- 已经废弃的方向
- 关键文件和链接
- 下一步只需要看的内容
```

然后开新会话继续。

这不是浪费上下文。

这是给 agent 换一张干净桌子。

什么时候该换？

我一般看四个信号：

```text
方向变了
agent 开始引用旧结论
会话里堆了大量日志和失败尝试
你已经说不清它现在到底该听哪条指令
```

出现这些信号，就别再养会话了。

长上下文像仓库，不是垃圾桶。

仓库要有货架，有标签，有过期清理。垃圾桶只会越来越满。

这件事不只影响 AI 编程。

用 agent 写公众号、做标书、整理访谈、做数据表、做产品图、做课程大纲，全都一样。你给它的不是更多字，而是一个工作现场。

现场乱，它就乱。

正确的用法很朴素：

```text
稳定的，写进文件。
临时的，用完就删。
复杂的，阶段性交接。
换方向，开新会话。
```

很多人以为自己缺的是更强模型。

有时候缺的只是把上下文收拾干净。
