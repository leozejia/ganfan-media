# 重生之我开中转站：开站很容易，这才是问题

营销号说的话，有一半是真的。

开中转站，技术门槛真的不高。GitHub 上找 new-api、sub2api 这类开源框架，部署一个实例，接上上游账号，换上自己的域名，基本就能开始卖余额了。不需要从头写协议，不需要自己做模型，甚至不需要完全搞懂 API 是怎么工作的。这部分没什么争议。

但他们漏掉了另一半。

---

漏掉的第一件事是获客。

营销号本人往往有粉丝基础，用户会主动找上来，他觉得开站容易，因为他自己不需要付获客的成本。

对大多数人来说，这个条件不存在。中转站市场已经很拥挤，各家都在打价格战，没有已有受众的话，凭什么让用户选你？获客才是这个生意里最贵的部分，而这块成本营销号没有算进去，因为他不需要。

所以他说的那个「容易」，只对有粉丝基础的人成立。

---

漏掉的第二件事是运维。

开站容易，维护是另一回事。

上游一变，你就要跟。不是合一下代码，是拆弹——这段能吸收，那段要延后，那段看起来很香但合进去会把你自己的服务状态搞坏。上游一次版本更新可能同时碰到请求字段、stream 事件、工具调用格式、usage 计费口径，任何一处没处理对，用户不会说「协议有问题」，他只会说「这个模型坏了」。

用户账单贵了来问，你得能拆开给他看：cache read 的计费口径和普通 input 不一样，stream 如果没有 terminal usage 要留审计日志，失败请求有没有产生上游成本要有说法。回答不出来，账单问题就会变成信任问题。

生产发布要有检查流程。一条 compose 命令可以让容器用不同的存储配置跑起来，数据库从 bind mount 切换到空卷，表现出来就是数据库「消失」了，但容器 healthy，页面能开，健康检查全通过。靠手感根本发现不了。

这些东西，不是几天能学会的。

SorryCode 也是拿 sub2api 搭起来的。开源框架部署快，但 bug 一直有。上个月追 GPT-5.5 和 image2 支持那段时间，连着熬了好几天，日均睡眠只有两三个小时。上游在动，image2 的流式路径要重新处理，账单口径要重新对，用户在用，服务不能停，代码还没稳。

那会儿是上线第二周，也让我终于知道那些做的好的中转站老板为什么每天都不睡觉了。

---

非技术背景的人开中转，大概率不是赚不到钱，而是先坑了用户。

不是说他们坏，是承诺了超出维护能力的事。服务不稳，账单说不清，出了问题没有人能处理，口碑就这么烂掉了。有粉丝基础解决了获客，解决不了运维。风险是站长扛的，技术债也是，用户只是用，出了事找你。

所以能把中转做好的，往往是对运维成本有清醒认知的技术背景的人——不是因为技术很难，而是因为你得知道东西坏了的时候去哪里找。

---

SorryCode 一直在做两件事：把运维那层做扎实，同时让普通人用起来省心。文档和一键安装有用户说做得好，不是因为什么都想到了，确实需要一些心思和耐心去迭代。
