# Claude Code 里跑 GPT，最怕的不是跑不起来

最怕的是它跑得起来。

做 SorryCode 中转的时候，我看到 sub2api 有个设置，可以让 GPT 跑在 Claude Code 里面。

第一眼看很酷。

Claude Code 的体验，GPT 的模型。

但这个组合有个问题：它太容易让人误会。

用户会以为自己只是换了个模型。

实际换掉的是整条 agent 调用链路。

我见过一个用户这么跑，一天烧掉几十块。账单很吓人，产出却很迷惑：项目没推进多少，肉眼可见的输出也不多。

这时候骂模型贵，骂不到点上。

问题不是 GPT 贵。

问题是 GPT 被放进了一个不一定适合 GPT 的 agent runtime 里。

Claude Code 当然可以跑 GPT。

中转层把 Claude Code 的请求改写成 OpenAI 那边能理解的格式，再把结果包回来。

能回消息。

能执行任务。

看起来一切正常。

但代码 agent 的成本和质量，不只看模型名字。

还要看两件事：

```text
上下文怎么组织
重复上下文怎么复用
```

代码 agent 每一轮都要带很多东西进去：

```text
历史
文件
diff
工具
系统提示
压缩记录
```

这些东西有些是资产。

比如稳定的项目规则、工具定义、长期不变的背景。

这些内容能被缓存复用，是好事。

但有些是垃圾。

比如已经失效的判断、走错方向的尝试、过期的错误信息、前几轮为了排障临时塞进去的上下文。

这些内容如果一直留在会话里，就会把 agent 带偏。

缓存命中不是魔法。

官方文档里讲得很清楚：prompt caching 只是复用模型已经处理过的 prompt 前缀，降低延迟和成本。它不会改变模型最终生成什么。

换句话说，缓存不会让脏上下文变干净。

它只会让重复上下文变便宜。

所以长会话不是越养越好。

一个会话养起来，可能发生两件事：

```text
稳定上下文被复用，成本下降
垃圾历史被保留，状态变脏
```

这就是双刃剑。

很多人只看到前半句。

长期跑代码任务，最理想的状态不是“会话越长越好”，而是：

```text
该稳定的东西稳定
该丢掉的东西丢掉
```

Codex 跑 GPT，是顺着这条路设计的。

GPT 是它的主场。

它怎么组织上下文、怎么复用前缀、怎么管理代码任务里的历史，整体更贴近 GPT 路径。

所以 GPT 做代码，默认先用 Codex，这是更自然的路径。

Claude Code 桥接 GPT，就不是这条默认路。

Claude Code 本来按 Claude / Anthropic 的方式组织上下文。

桥到 GPT 时，中间可能要改写消息结构、工具结构、历史结构。

这些改写不一定有恶意。

只是为了跑通。

但缓存最怕结构变化。

更怕你把一堆没用的历史继续带着跑。

能跑，只说明桥通了。

不说明它省钱。

也不说明它适合长任务。

所以这篇只想讲一个很小的判断：

```text
GPT 做任务，默认先用 Codex。
Claude 做任务，默认先用 Claude Code。
```

OpenClaw、Hermes Agent、Claude Code 都可以通过自定义 endpoint 去跑 GPT。

可以玩，但那是进阶玩法。

但我认为这是你账单变多的根本原因。。。

以后看到“某某 agent 也能跑 GPT”，先别兴奋。

先问两个问题：

```text
它能不能高效复用稳定上下文？
它会不会把垃圾历史一直带着跑？
```

这两个问题，比“能不能回消息”重要。

AI 工具现在最容易误导人的地方，就是把“能跑”包装成“适合”。

这两个词差得很远。

缓存命中也一样。

缓存命中高，不等于会话健康。

它只说明有一段东西被重复利用了。

至于那段东西是资产，还是垃圾，要你自己判断。
